# Детальный план реализации

Документ расширяет краткий чек-лист из `README.md` → раздел **План разработки**.
Здесь описаны цели, теоретическая база и конкретные шаги каждого этапа.

---

## 1. Введение и постановка задачи

Проект направлен на создание программной системы, способной в реальном
времени анализировать видеопоток и распознавать на нём объекты заданных
классов — людей, автомобили, спецтехнику, элементы средств индивидуальной
защиты (каски, жилеты) и другие. Система должна работать с частотой кадров,
достаточной для принятия оперативных решений: контроль пропускного режима,
мониторинг соблюдения техники безопасности, учёт транспорта на территории
предприятия.

Архитектура решения проектируется универсальной: замена модели или
дообучение на новом датасете позволяет адаптировать систему под любую
область. Ключевой конвейер обработки (захват видео, инференс модели,
постобработка, визуализация) остаётся неизменным, а вариативность
обеспечивается исключительно выбором модели и набором распознаваемых
классов.

### Сценарии применения

- **Контроль пропускного режима.** Распознавание государственных номерных
  знаков и сопоставление с белым списком; принятие решения о пропуске.
  Требует интеграции OCR поверх базовой детекции транспорта.
- **Мониторинг техники безопасности.** Детекция наличия каски, жилета,
  рукавиц; фиксация нарушений и алерты. Требует дообучения на датасете СИЗ.
- **Обнаружение людей и транспорта.** Подсчёт людей в зоне наблюдения,
  детекция ТС, отслеживание перемещений. Базовый сценарий «из коробки» на
  предобученной YOLO/COCO.
- **Обработка нефтегазовых данных.** Выявление разливов, утечек,
  несанкционированного доступа. Самый специализированный сценарий, требующий
  сбора уникального датасета и полного дообучения.

## 2. План реализации

### 2.1. Выбрать подходящую версию YOLO

Семейство YOLO включает несколько актуальных версий (YOLOv8, YOLOv9, YOLOv10,
YOLO11), каждая со своим балансом скорости инференса и точности. Выбор
определяется характером задачи, оборудованием и требованиями к FPS.

**Рекомендация для старта:** `yolov8s.pt` (small) — хороший баланс точности и
скорости; для embedded/CPU — `yolov8n.pt` (nano).

### 2.2. Изучить теоретическую базу

- **Архитектура YOLO.** Однопроходный детектор; сетка разбиения изображения;
  предсказание bounding box, confidence score и класса для каждой ячейки.
- **Bounding box и IoU.** Формат координат (`x_center, y_center, width,
  height`); метрика Intersection over Union; порог IoU в NMS.
- **Confidence score и порог уверенности.** Влияние `conf` на баланс ложных
  срабатываний и пропусков; эмпирический подбор под задачу.
- **Non-Maximum Suppression (NMS).** Устранение дублирующихся рамок; роль
  порога IoU.
- **mAP (mean Average Precision).** Различие `mAP@0.5` и `mAP@0.5:0.95`;
  использование при валидации и сравнении версий.
- **Transfer learning и fine-tuning.** Заморозка слоёв; выбор числа эпох и
  learning rate; почему fine-tuning эффективнее обучения с нуля.

### 2.3. Подготовить среду разработки

- Создать виртуальное окружение (venv или conda) с Python 3.9–3.11.
- Установить PyTorch: при наличии GPU — версию с CUDA (`torch+cu118` или
  `cu121`), при отсутствии — CPU-версию. Проверить `torch.cuda.is_available()`.
- Установить `ultralytics` — подтянет `opencv-python`, `Pillow` и др.
- Установить `opencv-python` и `matplotlib`.
- Проверить: `yolo predict model=yolov8s.pt source=0` — должно открыться окно
  с детекцией с веб-камеры.
- При работе с GPU: установить драйверы NVIDIA и CUDA Toolkit; `nvidia-smi`
  для мониторинга.

### 2.4. Реализовать базовый конвейер: захват видео и инференс

- Скрипт захвата кадра из источника и его отображения (без модели) —
  убедиться, что OpenCV корректно читает видеопоток.
- Добавить вызов `model(frame)` и вывод `results[0].plot()` — на экране
  появятся bounding boxes с классами и confidence.
- Измерить FPS: счётчик кадров и замер через `time.time()` или
  `cv2.getTickCount()`. Зафиксировать базовое значение для последующей
  оптимизации.

### 2.5. Настроить фильтрацию по классам и пороги уверенности

- Определить целевые классы: `person, car, truck, bus` — для базового
  сценария; добавить `helmet, vest` — после дообучения. Идентификаторы —
  в `model.names`.
- Реализовать фильтрацию: после инференса пройти по `results[0].boxes`,
  проверить `cls` и `conf`, оставить только целевые классы с `conf >= порога`.
- Подобрать порог: начать с `conf=0.25`, повышать до `0.4–0.5`, наблюдая за
  числом детекций.
- Настроить порог IoU для NMS (параметр `iou`): по умолчанию `0.7`, при
  сильном перекрытии снизить до `0.45–0.5`.

### 2.6. Подключить датасет и дообучить модель на кастомных классах

- **Найти датасет.** Roboflow Universe (`roboflow.com/universe`) — поиск по
  `helmet`, `PPE`, `safety vest`, `hard hat`. Экспорт в формате YOLOv8.
  Альтернативы: CVAT, Label Studio для самостоятельной разметки.
- **Подготовить данные.** Структура: `datasets/` → `images/` (`train/`,
  `val/`) и `labels/` (`train/`, `val/`). Каждый image-файл сопровождается
  `.txt` с координатами в формате YOLO. Создать `data.yaml` с путями и
  списком классов.
- **Запустить дообучение.** `yolo train model=yolov8s.pt data=data.yaml
  epochs=20 imgsz=640`. Результат — `best.pt` и `last.pt` в
  `runs/detect/train/`.
- **Валидация.** `yolo val model=best.pt data=data.yaml` — вывод mAP. Если
  mAP целевого класса ниже `0.6` — увеличить число эпох или объём датасета.
- Интеграция: `model = YOLO("best.pt")` — остальная логика без изменений.

### 2.7. Интегрировать модуль распознавания номерных знаков

- **Выбрать OCR.** EasyOCR — прост в установке, поддерживает кириллицу и
  латиницу; PaddleOCR — выше скорость и точность, но сложнее. Для старта —
  EasyOCR.
- **Пайплайн.** YOLO детектирует `car/truck/bus` → из bounding box
  вырезается область номера (нижняя треть задней части, либо отдельный класс
  `license plate`) → область передаётся в OCR → текст нормализуется (удаление
  пробелов, верхний регистр).
- **Альтернатива.** Дообучить YOLO на класс `license plate` для точной
  локализации перед OCR — повышает точность распознавания.

### 2.8. Реализовать логику принятия решений

- **Белый список номеров.** Словарь/файл (JSON/CSV) с разрешёнными номерами.
  Поиск по списку: совпадение — допуск, отсутствие — алерт. Визуально:
  зелёная рамка для допущенных, красная для неразрешённых.
- **Детекция нарушений СИЗ.** Человек без каски/жилета → алерт с timestamp,
  координатами и сохранением кадра-доказательства. Порог уверенности для
  нарушения — `0.5–0.6`.
- **Логирование.** Запись каждого события в CSV/JSON: время, тип, класс,
  confidence, путь к кадру. Позволяет анализировать работу постфактум и
  строить дашборды.

### 2.9. Оптимизировать производительность

Цель — не менее 10–15 FPS для работы в реальном времени.

- **Уменьшение разрешения входа.** `imgsz`: снижение с 640 до 416 или 320
  ускоряет обработку ценой качества детекции мелких объектов.
- **Half-precision (FP16).** `model(frame, half=True)` — сокращение вычислений
  на GPU примерно вдвое с минимальной потерей точности. На CPU эффекта не даёт.
- **Экспорт в ONNX.** `yolo export model=best.pt format=onnx` + инференс через
  `onnxruntime`. Прирост скорости 1.5–2x.
- **Экспорт в TensorRT.** При наличии NVIDIA GPU — `yolo export model=best.pt
  format=engine`. Ускорение до 3–5x.
- **Пропуск кадров.** При 30 FPS источника и 10 FPS обработки — обрабатывать
  каждый третий кадр, промежуточные пропускать.
